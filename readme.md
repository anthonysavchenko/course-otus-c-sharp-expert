## 1. Разработка ядра хранилища и zero-allocation парсера

**Лекция 18.05.2026 Низкоуровневая работа с памятью: Span<T>, Memory<T>, stackalloc**

### Цель

- Применить на практике знания о низкоуровневой работе с памятью для создания высокопроизводительного компонента.
- Разработать два фундаментальных, независимых модуля, которые станут основой для всех последующих домашних заданий.
- Заложить архитектурный и технологический фундамент для курсового проекта, решая аналогичные задачи в упрощенном виде.

### Результат

Студент создаст два готовых к интеграции компонента:
- Базовое хранилище SimpleStore: Класс, отвечающий за хранение данных в памяти.
- Высокопроизводительный парсер CommandParser: Статический класс, способный разбирать входящие команды без выделения дополнительной памяти в управляемой куче (zero-allocation).

### Зачем это нужно

Это задание учит подходить к проектированию производительных систем с самого начала, а не пытаться оптимизировать их постфактум. Навык написания zero-allocation кода является ключевым для разработки высоконагруженных сетевых сервисов, и это ДЗ дает возможность отработать его на изолированной, понятной задаче.

### Описание/Пошаговая инструкция выполнения домашнего задания

Пункт 1. Реализация базового хранилища SimpleStore
Создайте класс SimpleStore. Внутри он должен использовать Dictionary<string, byte[]> для хранения данных. Реализуйте три публичных метода:
void Set(string key, byte[] value): добавляет или обновляет значение по ключу.
byte[] Get(string key): возвращает значение по ключу или null, если ключ не найден.
void Delete(string key): удаляет ключ и значение.
На данном этапе потокобезопасность не требуется.

Пункт 2. Проектирование парсера команд
Создайте статический класс CommandParser. В нем создайте публичный статический метод Parse, который принимает ReadOnlySpan и возвращает структуру с тремя полями: (ReadOnlySpan Command, ReadOnlySpan Key, ReadOnlySpan Value). Этот метод будет отвечать за разбор команды.

Пункт 3. Реализация логики парсинга
Реализуйте логику метода Parse. Он должен разбирать входящую последовательность байт, представляющую собой строку вида "COMMAND KEY VALUE" (например, "SET user:1 data"). Команда, ключ и значение разделены пробелами. Необходимо использовать методы IndexOf и Slice для Span, чтобы извлечь три компоненты без создания новых строк или массивов.

Пункт 4. Обработка команд с разным числом аргументов
Модифицируйте парсер так, чтобы он корректно обрабатывал команды с разным количеством аргументов. Например, команда GET user:1 содержит только команду и ключ. В этом случае Value в возвращаемой структуре должен быть пустым (.IsEmpty). Если команда некорректна (например, отсутствует ключ), метод должен возвращать пустую структуру по умолчанию.

Пункт 5. Написание Unit-тестов для парсера
Создайте проект с юнит-тестами на xUnit. Напишите тесты для CommandParser, которые проверяют следующие сценарии:
Корректный разбор команды SET с тремя аргументами.
Корректный разбор команды GET с двумя аргументами.
Обработка некорректной команды (например, без ключа).
Обработка команды с лишними пробелами между аргументами.

### Критерии оценки

Пункт 1 - 1 балл
Пункт 2 - 2 балла
Пункт 3 - 3 балла
Пункт 4 - 2 балла
Пункт 5 - 2 балла

Для зачёта домашнего задания достаточно 8 баллов.

## 2. Каркас TCP-сервера и интеграция парсера

**Лекция 25.05.2026. Работа с неуправляемыми ресурсами: IDisposableAsync, SafeHandle, NativeMemory**

### Цель

- Получить практический опыт создания асинхронного сетевого сервера с использованием System.Net.Sockets.
- Научиться эффективно управлять памятью при сетевом вводе-выводе с помощью ArrayPool.
- Интегрировать независимые компоненты (сетевой уровень и парсер из ДЗ №1) в единую систему.
- Освоить базовые принципы обработки жизненного цикла клиентского подключения.

### Результат

Студент создаст консольное приложение, которое представляет собой работающий TCP-сервер. Этот сервер способен:
- Принимать входящие подключения от клиентов.
- Асинхронно читать данные из сокета, используя буферы из пула.
- Передавать полученные данные в CommandParser из предыдущего ДЗ.
- Выводить результат парсинга в консоль.
- Корректно обрабатывать отключение клиента.

### Зачем это нужно

Это задание является критически важным шагом от создания изолированных компонентов к построению полноценного сетевого приложения. Студент на практике столкнется с задачами асинхронного I/O, управления ресурсами и интеграции, которые являются основой для разработки любого backend-сервиса.

### Описание/Пошаговая инструкция выполнения домашнего задания

Пункт 1. Создание класса TCP-сервера
Создайте класс TcpServer. В нем реализуйте метод StartAsync, который будет инициализировать Socket, связывать его с локальным IP-адресом и портом (например, 127.0.0.1:8080) и переводить в режим прослушивания (Listen).

Пункт 2. Реализация цикла приема подключений
В методе StartAsync после вызова Listen организуйте бесконечный асинхронный цикл (while(true)), который ожидает новые подключения с помощью await serverSocket.AcceptAsync(). Для каждого принятого клиентского сокета запускайте отдельную задачу для его обработки (например, _ = ProcessClientAsync(clientSocket)).

Пункт 3. Чтение данных от клиента и парсинг
Реализуйте приватный асинхронный метод ProcessClientAsync(Socket clientSocket). Внутри него организуйте цикл для чтения данных. При чтении (await clientSocket.ReceiveAsync(...)) используйте буфер, арендованный из ArrayPool.Shared. Полученные данные (в виде ReadOnlyMemory) передавайте в статический метод CommandParser.Parse из ДЗ №1. Результат парсинга (команду, ключ, значение) выводите в консоль. Не забудьте возвращать буфер в пул после использования.

Пункт 4. Обработка отключения клиента
Модифицируйте цикл чтения данных в ProcessClientAsync. Если вызов ReceiveAsync возвращает 0, это означает, что клиент закрыл соединение. В этом случае необходимо прервать цикл, корректно закрыть сокет клиента (Shutdown, Close, Dispose) и завершить задачу обработки.

Пункт 5. Запуск сервера
В Program.cs создайте экземпляр вашего TcpServer, вызовите его метод StartAsync и обеспечьте работу приложения в фоновом режиме, чтобы оно не завершилось сразу после запуска (например, с помощью Console.ReadLine()).

### Критерии оценки

Реализован класс TcpServer с методом StartAsync, который корректно инициализирует и запускает прослушивающий сокет - 1 балл
Реализован асинхронный цикл приема подключений, который для каждого нового клиента запускает отдельную задачу обработки - 2 балла
Реализован цикл чтения данных от клиента, который использует буферы из ArrayPool и передает данные в CommandParser из ДЗ №1 - 4 балла
Корректно обрабатывается отключение клиента (обнаружение, закрытие сокета, выход из цикла) - 2 балла
Создан исполняемый файл Program.cs, который успешно запускает сервер и позволяет к нему подключиться (например, через telnet) - 1 балл

## 3. Потокобезопасное хранилище и сбор статистики

**Лекция 04.06.2026. Каналы (System.Threading.Channels): современный асинхронный producer/consumer**

### Цель

- Применить на практике продвинутые примитивы синхронизации для создания высокопроизводительных потокобезопасных структур данных.
- Освоить lock-free техники для сбора метрик в многопоточной среде, избегая узких мест производительности.
- Понять разницу в производительности между блокирующими и lock-free подходами при решении разных задач.

### Результат

Студент получит модифицированный класс SimpleStore из ДЗ №1, который теперь можно безопасно использовать из нескольких потоков одновременно. Этот класс также будет предоставлять публичные свойства для получения статистики своей работы (количество операций).

### Зачем это нужно

Любой серверный сервис по своей природе является многопоточным. Умение правильно защищать разделяемые данные — это фундаментальный навык. Это ДЗ учит использовать не просто lock, а более тонкий и производительный инструмент ReaderWriterLockSlim, а также показывает, как избежать блокировок вообще с помощью Interlocked там, где это возможно, что является признаком экспертного подхода.

### Описание/Пошаговая инструкция выполнения домашнего задания

Пункт 1. Возьмите класс SimpleStore из ДЗ №1. Добавьте в него приватное поле private readonly ReaderWriterLockSlim _lock = new ReaderWriterLockSlim();. Модифицируйте методы Set, Get и Delete так, чтобы они использовали этот лок. Get должен использовать блокировку на чтение (EnterReadLock/ExitReadLock), а Set и Delete — блокировку на запись (EnterWriteLock/ExitWriteLock). Обязательно используйте конструкцию try...finally для гарантированного освобождения блокировки.

Пункт 2. Добавление полей для статистики
Добавьте в SimpleStore три приватных long поля для хранения статистики: _setCount, _getCount, _deleteCount. Создайте публичный метод или свойство (например, GetStatistics()), которое будет возвращать кортеж (long setCount, long getCount, long deleteCount).

Пункт 3. Реализация сбора статистики
Модифицируйте методы Set, Get и Delete. Внутри каждого метода после успешного выполнения основной операции инкрементируйте соответствующий счетчик. Используйте Interlocked.Increment(ref _fieldName) для потокобезопасного увеличения счетчиков.

Пункт 4. Реализация IDisposable
Реализуйте в классе SimpleStore интерфейс IDisposable. В методе Dispose освободите ресурсы, занимаемые ReaderWriterLockSlim (вызвать его метод Dispose).

Пункт 5. Написание тестов на многопоточность
Создайте новый юнит-тест (или добавьте в существующий). Напишите тест, который запускает несколько задач (Task.Run) параллельно. Часть задач должна вызывать метод Set, а другая часть — Get. Тест должен проверить, что после выполнения всех задач данные в хранилище корректны и не произошло гонки состояний, а также что счетчики статистики показывают правильные значения.

### Критерии оценки

Метод Get использует ReaderWriterLockSlim в режиме чтения (EnterReadLock/ExitReadLock) - 2 балла
Методы Set и Delete используют ReaderWriterLockSlim в режиме записи (EnterWriteLock/ExitWriteLock), блокировки корректно освобождаются в блоке finally - 3 балла
Реализован потокобезопасный сбор статистики с использованием Interlocked.Increment для трех операций - 2 балла
Класс SimpleStore реализует интерфейс IDisposable и корректно освобождает ресурсы ReaderWriterLockSlim - 1 балл
Написан многопоточный тест, который доказывает корректность работы хранилища и сбора статистики под нагрузкой - 2 балла
